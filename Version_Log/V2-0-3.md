# V2.0.3 变更记录与证明更新指南

本文档总结新 `symexec` / SAC、`AggressivePreProcess`、`SumLib` 与 `minigmp_sumlib` 例子带来的主要修改点，并整理证明失效后的更新方法。

## 变化概览

- Windows 二进制工具完成更新，包括 `win-binary/symexec.exe`、`mcp.exe`、`lsp.exe`、`StrategyCheck.exe`，证明流水线文档同步补充 Windows 编译循环约束。
- `AggressivePreProcess` 使大量 `*_goal.v` 重新生成，`pre_process` 展开的上下文结构发生变化，旧证明中的 `H` / `H0` / `H1` 等编号经常变成 `PreH1` / `PreH2` / ...
- 一些 disjunction 目标的分支顺序或目标形状变化，导致原来的 `Left` / `Right` 选择失效。
- 新增 `SeparationLogic/SumLib`，包括 `Sum.v`、`ZRange.v`、`ZRect.v`、`ListMember.v`、`GraphSum.v` 等，并加入 `_CoqProject` / `SeparationLogic/_CoqProject` 逻辑路径。
- 新增并刷新 `minigmp_sumlib` 例子，包括 C 侧文件、Rocq 侧文件、Windows / Linux symexec 脚本入口和生成目标。
- `minigmp_sumlib` 基于新的 Windows 脚本和 SAC 重新执行，刷新 `gmp_goal.v` / `gmp_proof_auto.v`，并修复 `gmp_proof_manual_part1.v` 到 `gmp_proof_manual_part6.v`。
- `QCP_demos_LLM/sll_merge_rel` 和 `sll_split_while` 在新生成目标下完成 annotation / manual proof 修复。

## 主要修改点

### 1. Windows 脚本成为正式刷新入口

`minigmp_sumlib` 使用 `run-example-windows.sh` 中的命令重新运行：

```powershell
win-binary/symexec.exe --goal-file=SeparationLogic/examples/Applications_human/minigmp_sumlib/gmp_goal.v --proof-auto-file=SeparationLogic/examples/Applications_human/minigmp_sumlib/gmp_proof_auto.v --proof-manual-file=SeparationLogic/examples/Applications_human/minigmp_sumlib/gmp_proof_manual.v -IQCP_examples/Applications_human/minigmp_sumlib/ -IQCP_examples/Applications_human/ -slp QCP_examples/Applications_human/ SimpleC.EE.Applications_human -IQCP_examples/QCP_demos_human/ -slp QCP_examples/QCP_demos_human/ SimpleC.EE.QCP_demos_human --coq-logic-path=SimpleC.EE.Applications_human.minigmp_sumlib --input-file=QCP_examples/Applications_human/minigmp_sumlib/gmp.c --no-exec-info
```

注意：

- `-I` 用于 C 头文件搜索路径。
- `-slp` 用于生成 / 定位 Rocq 逻辑路径。
- 两者不能互相替代。
- `symexec` 不会覆盖已经存在的 `*_proof_manual.v`，出现 `manual proof file not updated` 警告是正常现象。

### 2. SumLib 改变了 `list_to_Z` 的化简行为

`minigmp_sumlib` 的若干证明从原来的列表递归定义迁移到 `SumLib.Sum.sum` 形式后，旧证明里常见的：

```coq
simpl list_to_Z. lia.
```

不再总能工作。典型失败信息是：

```text
Tactic failure: Cannot find witness.
```

或目标中出现：

```coq
Sum.sum (fun i => 0 <= i < Zlength (...)) ...
```

更新方法：

- 对单元素列表优先使用 `list_to_Z_single`。
- 不要过早 `unfold list_to_Z`，否则会暴露 `Sum.sum` 目标，`lia` 无法自动构造区间 witness。
- 对 append 场景继续优先使用现有 lemma：
  - `list_to_Z_concat`
  - `list_to_Z_concat_r`
  - `list_to_Z_list_append`
  - `Zlength_app`
  - `Zlength_sublist`

示例：

```coq
rewrite !list_to_Z_single.
lia.
```

如果 `sublist` 展开后右侧仍有 `list_to_Z UINT_MOD (z :: nil)`，需要在 `destruct` 后再补一次：

```coq
rewrite list_to_Z_single.
lia.
```

### 3. `pre_process` 后 hypothesis 编号漂移

新 SAC / `AggressivePreProcess` 后，旧证明中的普通 `H` 编号经常变化为 `PreH` 编号。常见修复方式：

- 先查看对应 `*_goal.v` 中 witness theorem 的 forall 参数。
- 用 `Show.` 临时打印当前目标和 context，确认需要的 hypothesis 名称。
- 把旧证明中的 `H3` / `H5` / `H11` 等替换成当前的 `PreH5` / `PreH7` / `PreH13` 等。
- 修完后必须删除所有临时 `Show.`。

推荐写法是尽量减少硬编码编号：

```coq
match goal with
| Hlen : Zlength l = _ |- _ => rewrite Hlen
end.
```

这比直接写 `rewrite H12` 更能抵抗后续生成器重排。

### 4. disjunction 分支可能反转

一些 return witness 的目标中，旧证明的：

```coq
Left. Left.
```

可能需要改成：

```coq
Right.
```

反之亦然。判断方法不是照抄旧证明，而是看当前 goal 的 disjunction 结构。例如 `mpn_cmp` / `gmp_cmp` 一类目标，需要根据当前目标表达的是：

- `a > b`
- `a = b`
- `a < b`

来选择正确分支。

### 5. 高位为空时要显式化简 nil 部分

在 `mpn_add_return_wit_1` 这类边界 case 中，新目标可能保留：

```coq
val_a = val_a_low + list_to_Z UINT_MOD nil * UINT_MOD ^ bn_pre
```

此时需要先把 nil 高位化掉：

```coq
simpl in PreH12.
replace (val_a_low + 0 * UINT_MOD ^ bn_pre) with val_a_low in PreH12 by lia.
subst val_a.
```

然后再交给 `entailer!` 或 `lia`。

### 6. 结构字段记号要保持 UTF-8 正确

`minigmp` / `minigmp_sumlib` 中 struct 字段访问使用：

```coq
&( r_pre # "__mpz_struct" ->ₛ "_mp_alloc")
```

迁移或复制文件时，如果编码出错，可能变成乱码，例如 `->ₛ` 被破坏成错误字符。这会导致语法错误：

```text
Syntax error: ')' expected after [custom:lvalue_expr_entry level 99]
```

修复方法是恢复为 `->ₛ`，并确保文件以 UTF-8 保存。不要把该记号替换成普通 ASCII `->`。

## 其他人更新证明的推荐流程

### Step 1: 先刷新生成文件

进入仓库根目录，运行对应脚本中的 `symexec` 命令。对 `minigmp_sumlib`，使用 Windows 命令见上文。

刷新成功的标准：

- 输出出现 `Successfully finished symbolic execution`。
- 每个目标函数都有 `End of symbolic execution of function ...`。
- `*_goal.v` / `*_proof_auto.v` 被更新。
- 已存在的 `*_proof_manual.v` 不被覆盖是正常行为。

### Step 2: 先编译依赖库

`minigmp_sumlib` 需要先保证 SumLib 与辅助库可编译：

```powershell
cd SeparationLogic
make SumLib/Sum.vo SumLib/ZRange.vo SumLib/ZRect.vo SumLib/ListMember.vo SumLib/GraphSum.vo SumLib/SumLib.vo
make examples/Applications_human/minigmp_sumlib/GmpAux.vo
make examples/Applications_human/minigmp_sumlib/GmpNumber.vo
make examples/Applications_human/minigmp_sumlib/GmpMulAux.vo
```

Windows 下 `GmpAux.vo` / `GmpNumber.vo` 可能耗时较长。如果命令超时，要先检查 `coqc` 是否仍在后台运行，不要立刻重复启动多个编译。

### Step 3: 分 part 修 manual proof

逐个编译：

```powershell
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part1.vo
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part2.vo
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part3.vo
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part4.vo
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part5.vo
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual_part6.vo
```

修复顺序建议：

1. 先处理语法错误和编码错误。
2. 再处理 `The reference H... was not found`，改为当前 `PreH...` 或 `match goal`。
3. 再处理 `Attempt to save an incomplete proof`，用临时 `Show.` 找剩余目标。
4. 最后处理 `Cannot find witness`，通常是 `Sum.sum` / `list_to_Z` 化简问题。

### Step 4: 汇总编译和 goal check

```powershell
make examples/Applications_human/minigmp_sumlib/gmp_proof_manual.vo
make examples/Applications_human/minigmp_sumlib/gmp_goal_check.vo
```

通过后再重新运行一次 `symexec`，确认生成文件仍最新，然后再跑：

```powershell
make examples/Applications_human/minigmp_sumlib/gmp_goal_check.vo
```

### Step 5: 最终文本审计

最终至少检查：

```powershell
rg -n "\bShow\.|Admitted\.|\bAxiom\b" SeparationLogic/examples/Applications_human/minigmp_sumlib -g "gmp_proof_manual*.v" -g "Gmp*.v"
```

期望没有输出。若有：

- `Show.` 必须删除。
- `Admitted.` 不允许遗留。
- 新增 `Axiom` 不允许出现在 manual proof 或 case-local formal lib 中。

还应检查是否有后台编译进程：

```powershell
Get-Process | Where-Object { $_.ProcessName -match 'coqc|mingw32-make|make|symexec' }
```

## win-binary VC 打印更新后的证明修复经验

本轮 Windows binary 更新后，VC 打印更加细化，多个 case 的 witness 编号、return witness 归属和 disjunction 分支顺序发生了变化。修复时不要按旧 `proof_of_*_wit_N` 的编号机械搬运证明，应先打开对应 `*_goal.v`，确认当前 witness 的 `forall` 参数、`PreH` 语义和左右分支含义。

典型模式：

- `*_return_wit_1` 可能已经被 `*_proof_auto.v` 自动证明。若 `goal_check` 报 `label ... is already declared`，应删除 manual 中同名 theorem，并把旧证明体迁移到真正缺失的 witness，例如 `int_array_merge_rel` 中 `mergeSort_return_wit_1/2` 的错位。
- 相邻分支可能整体对调。`sortArray_entail_wit_4_1/4_2`、human `sortArray_entail_wit_6_1/6_2` 分别出现了 `j < 0` 头插 case 与 `j >= 0` 中间插入 case 对调；应按当前目标中的 `j < 0`、`Znth ... <= key`、`strict_upperbound` 等语义交换 lemma 声明或证明体。
- `PreH` 编号不能当作稳定接口。优先用语义名重建局部事实，例如把 `IntArray.seg_Zlength` 的结果显式命名为 `Hlen_tail` / `Hlen_l4`，把旧 `H8`、`H10` 改为当前目标中的 `PreH8`、`PreH11`，或直接用 `match goal` 查找长度等式。
- 空段证明不要依赖旧版显式等式。若新版目标只给出 `i >= q + 1` 和 `IntArray.seg arr i (q + 1) l`，先用 `IntArray.seg_length` / `IntArray.seg_Zlength` 推出 `l = nil` 或端点相等，再用 `IntArray.seg_empty`。`int_array_merge_rel` 的 `merge_entail_wit_5_1/5_2` 已改成这种写法。
- `lia` 报 `Cannot find witness` 时，常见原因是还处在 separation-logic entailment 外壳里，或目标中缺少显式长度化简。先用 `Intros_p` 命名纯事实、`Zlength_cons` / `Zlength_nil` / `Zlength_sublist` 显式化，再交给 `lia`；不要在 entailment 顶层硬写普通 `assert`。
- 对 `sublist_single`、`seg_empty`、`seg_Zlength` 这类库 lemma，参数顺序和目标形态要按当前库检查。human `sortArray` 中旧写法 `sublist_single j l2_2 0` 应改为 `sublist_single 0 j l2_2`。
- 编译慢的目标可以先逐个 `goal_check` 修复；`multiple_knapsack_goal_check.vo` 仍然耗时较长，若 PowerShell 超时，应先检查后台 `coqc` 是否还在运行，避免重复启动。

本轮已重新运行 `run-example-windows.sh` 刷新生成文件，并修复了受影响的 manual proofs；Makefile 可达的 changed `*_goal_check.vo` 已重新验证。`QCP_demos_human/sortArray_goal_check.v` 当前 Makefile 直接 target 未接上，使用 `_CoqProject` 同等 loadpath 按 `sortArray_lib.v -> sortArray_goal.v -> sortArray_proof_auto.v -> sortArray_proof_manual.v -> sortArray_goal_check.v` 顺序 direct `coqc` 验证通过。

## 常见错误与处理表

| 错误信息 | 常见原因 | 处理方式 |
| --- | --- | --- |
| `The reference H... was not found` | `pre_process` 后 hypothesis 编号漂移 | 看当前 goal，把旧 `H` 改成 `PreH`，或改用 `match goal` |
| `Attempt to save an incomplete proof` | `entailer!` 后还有纯目标未收掉 | 临时插入 `Show.`，针对剩余目标补 `rewrite` / `lia` |
| `Cannot find witness` | 展开了 `Sum.sum`，`lia` 找不到区间 witness | 避免 `unfold list_to_Z`，使用 `list_to_Z_single` 等 lemma |
| `Syntax error ... lvalue_expr_entry` | `->ₛ` 字段记号编码损坏 | 恢复 UTF-8 的 `->ₛ` |
| `Cannot find physical path bound to logical path SumLib` | SumLib 未编译或 `_CoqProject` 路径未生效 | 先编译 `SumLib/*.vo`，确认 `_CoqProject` 包含 SumLib 映射 |
| Windows `coqc` 超时 | 编译仍在后台运行 | 先查进程和 `.vo` 是否生成，避免重复启动 |

## VC branch and numbering drift repair notes

- After regenerating goals, do not trust old `proof_of_*_wit_N` numbering. Re-open the current `*_goal.v` and identify the semantic branch from its `forall` parameters and `PreH` facts. This repaired `exgcd_return_wit_2/3/4/5/6`, `sll_insert_sort` insertion branches, and BST delete return witnesses.
- If `goal_check` reports `label ... is already declared`, the witness is already supplied by `*_proof_auto.v`; remove the same-named manual lemma instead of proving it twice. `sll_insert_sort` now keeps manual proof only for the missing `insertion_entail_wit_3_2`.
- For swapped disjunction branches, move the whole proof body to the branch matching the current goal, then fix `PreH` names by meaning. Examples: `sll_insert_sort` moved the empty-tail proof to `insertion_return_wit_1` and the nonempty-tail proof to `insertion_return_wit_2`; `dll_queue` reordered empty/nonempty enqueue and dequeue returns.
- Avoid setoid `rewrite PreH...` inside large separation assertions when the hypothesis is an equality over a data structure. Prefer `subst <name>; simpl` or prove a local equality such as `Hdel` and rewrite before folding spatial predicates. This avoided typeclass evar explosions in BST delete proofs.
- For BST delete equal-key cases, choose the returned pointer from the postcondition, not from the old witness name: left-recursive/right-recursive branches return the original root; `left = 0` returns the right subtree; `left <> 0` combines the replacement node with `store_combine`.
- `bst_delete_rec2` and `bst_fp_delete` showed that adjacent recursive/equal-key branches can rotate as a block. Re-map `return_wit_2/3/4/5` from the current postcondition shape: recursive branches rebuild the original root, empty child branches return the sibling/zero tree, and replacement-min branches need the current `find_pre`/`delete_min` facts rather than old `PreH` numbers.
- In `fme`, several generated `||` goals changed branch order. Pick `Left`/`Right` from the pure discriminator in the goal (`res <> 0`, `retval = 0`, `0 = 0`, `1 = 1`, etc.), not from the old proof. For nested disjunctions, confirm the association by the expected pure formula: e.g. `lia_deduction_return_wit_1` reaches the third `0 = 0` branch with a single `Right`.
- When a manual witness is now proved by `*_proof_auto.v`, delete the manual lemma to avoid duplicate labels. `fme` needed this for `free_InequList_return_wit_2` and `real_shadow_return_wit_2/4`; missing auto witnesses such as `free_InequList_return_wit_3` and `real_shadow_return_wit_3/5` should be added manually.
- For goals that only weaken concrete pointer fields to existential/underscore fields, `entailer!` may leave a small spatial goal. Use `store_ptr_undef_store_ptr` after selecting the correct branch, and use `Show.` temporarily only to inspect the remaining entailment; remove it before final checking.
- For arithmetic safety VCs, adjacent witness numbers may swap between direct bound and triangle-bound obligations. In `exgcd`, the `x - q*y` bound needs `exgcd_reduction`, while the product-only bound can use `exgcd_reduction'` plus an explicit nonnegative upper bound.
- For StdLib migration, ensure manual proofs import the same logical library as generated goals. A stale import of `SimpleC.EE.LLM_bench.Engineering.string.string_lib` shadows `SimpleC.StdLib.string_lib` and makes `store_string` fail to cancel even if the definitions are textually identical.
