# V2.0.2 版本更新与证明迁移指南

本文档总结从 QCP V20260421 迁移到 QCP 2.0.2 过程中的主要变化，并给出使用者升级二进制、刷新生成文件、修复证明时的操作建议。

## 变化概览

- 生成文件迁移到 QCP V20260421 之后的新输出格式，LiteOS、QCP demos、strategy goal / proof 等目标发生批量刷新。
- 示例目录从旧命名迁移到 human / LLM 两套新命名，`run-example-linux.sh`、`run-example-windows.sh` 和脚本数据同步更新。
- C annotation 与 strategy 中的表示谓词从旧 `data_at` 风格迁移到新的 `store` / `poly_store` 风格。
- 纯断言记号从旧 `[| ... |]` 改为中文引号样式 `“ ... ”`。
- 新 SAC 下少量 manual proof 需要修补，典型变化是 hypothesis 编号、自动化展开后的目标形状、部分库 lemma 的使用方式变化。
- 新增字符串字面量支持和 `StringLib.v`，字符串相关 generated goal / auto proof 发生刷新。
- 三个平台的 QCP 2.0.2 二进制工具和 QIDE 扩展完成更新。

## 主要行为变化

### 1. 示例目录命名固定为 human / LLM 两套

升级后不要再使用旧目录：

```text
QCP_examples/Applications
QCP_examples/QCP_democases
QCP_examples/LLM_friendly_cases
```

应使用：

```text
QCP_examples/Applications_human
QCP_examples/QCP_demos_human
QCP_examples/QCP_demos_LLM
```

`SeparationLogic/examples` 下也对应使用：

```text
SeparationLogic/examples/Applications_human
SeparationLogic/examples/QCP_demos_human
SeparationLogic/examples/QCP_demos_LLM
```

迁移已有 case 时，重点检查：

- `#include` 路径。
- `run-example-*.sh` 中 `--input-file`、`--goal-file`、`--proof-auto-file`、`--proof-manual-file`。
- `--coq-logic-path` 是否与新的目录模块名一致。
- `-slp` 的物理路径和逻辑路径是否成对更新。

### 2. 表示谓词从 `data_at` 迁到 `store`

旧 annotation / strategy 中常见的 `data_at` 风格不再是推荐写法。升级后应使用当前 store 体系：

```coq
poly_store ...
```

或生成目标中的：

```coq
(&(...)) # Int |-> v
```

处理方式：

- 不要为了兼容旧证明而把新生成文件手动改回 `data_at`。
- C annotation 和 `.strategies` 应优先按新 `store` / `poly_store` 语义改写。
- manual proof 中若出现空间谓词匹配失败，先检查左右目标是否一个仍是旧 `data_at`、一个已是新 `store`。
- 对 LiteOS、链表、数组类 case，通常需要同步更新 shape strategy 中的字段 store 规则。

### 3. 纯断言记号从 `[| |]` 改为 `“ ”`

旧版目标中常见：

```coq
[| P |]
```

新版使用：

```coq
“ P ”
```

影响：

- `*_goal.v` 会大面积刷新。
- manual proof 中如果直接 rewrite / unfold 旧纯断言结构，可能失效。
- 自定义 helper lemma 若写死旧纯断言构造，需要改成新记号或底层当前定义。

建议：

- 对生成文件只重新运行 `symexec`，不要手工替换。
- 对 helper lemma，优先用现有 tactic，例如 `entailer!`、`pre_process`、`Intros`，减少直接操作纯断言内部构造。
- 文档和教程中的旧 `[| |]` 也应同步更新，避免新使用者照旧写 annotation。

### 4. 字符串字面量成为内建支持

`Support String Literal and port to new QCP` 引入了 `StringLib.v`，字符串相关 case 的生成目标明显变化。

影响较大的例子：

- `LLM_bench/Engineering/string/memchr`
- `memory`
- `search`
- `strcat`
- `strcmp`
- `strcpy`
- `strlen`

更新方法：

- 确认 `SeparationLogic/SeparationLogic/SeparationLogic.v` 已导入 `StringLib`。
- 重新运行对应 `symexec`，刷新 goal / auto。
- 字符串相关 manual proof 不要假设字符串字面量一定被展开成旧数组初始化形式。
- 如果目标中出现新的字符串库谓词或 lemma，优先在 `StringLib.v` 中寻找对应规则。

### 5. 新 SAC 会改变 VC 形状

从 V20260421 到 2.0.2 的过程中，`symexec` / SAC 多次更新，常见影响包括：

- generated `forall` 参数顺序变化。
- hypothesis 编号变化。
- `proof_auto` 中自动生成的子目标数量变化。
- strategy goal 的空间谓词顺序变化。
- 一些原本 auto 可解的目标转移到 manual，或反过来。

更新 manual proof 时，优先判断是哪一类变化：

- 如果只是 hypothesis 名称变了，改 proof，不改 annotation。
- 如果空间谓词顺序变了，优先用交换律 / 已有 `sep_apply` 规则，而不是改 generated goal。
- 如果 VC 语义明显不可证，再回到 annotation 或 strategy 修正。

## 使用者升级流程

### Step 1: 更新二进制和 QIDE

按平台确认以下文件来自 QCP 2.0.2：

```text
linux-binary/symexec
linux-binary/mcp
linux-binary/lsp
linux-binary/StrategyCheck

mac-arm64-binary/symexec
mac-arm64-binary/mcp
mac-arm64-binary/lsp
mac-arm64-binary/StrategyCheck

mac-x86-64-binary/symexec
mac-x86-64-binary/mcp
mac-x86-64-binary/lsp
mac-x86-64-binary/StrategyCheck

win-binary/symexec.exe
win-binary/mcp.exe
win-binary/lsp.exe
win-binary/StrategyCheck.exe
qide.vsix
```

Windows 用户使用 `win-binary/*.exe`，Linux 用户使用 `linux-binary/*`，macOS 用户按架构选择 `mac-arm64-binary` 或 `mac-x86-64-binary`。

### Step 2: 更新路径和脚本

对旧 case 做路径迁移：

```text
Applications -> Applications_human
QCP_democases -> QCP_demos_human
LLM_friendly_cases -> QCP_demos_LLM
```

脚本中同时检查：

- `--input-file`
- `--goal-file`
- `--proof-auto-file`
- `--proof-manual-file`
- `--coq-logic-path`
- `-I`
- `-slp`

注意：`-I` 是 C include 搜索路径，`-slp` 是 Rocq 逻辑路径映射，二者不能互相替代。

### Step 3: 迁移 annotation / strategy 语法

重点改三类：

1. 旧 `data_at` 风格改为 `store` / `poly_store`。
2. 旧 `[| P |]` 纯断言改为 `“ P ”`。
3. 字符串字面量相关 annotation 按 `StringLib` 和内建字符串规则处理。

迁移后先只跑单个 case 的 symexec，不要一次全仓库刷新。

### Step 4: 重新运行 symbolic execution

以 Windows 为例：

```powershell
win-binary/symexec.exe --goal-file=<..._goal.v> --proof-auto-file=<..._proof_auto.v> --proof-manual-file=<..._proof_manual.v> -I<include-dir> -slp <physical-dir> <logical.path> --coq-logic-path=<case.logical.path> --input-file=<case.c> --no-exec-info
```

成功标志：

- 每个函数出现 `End of symbolic execution of function ...`。
- 最后出现 `Successfully finished symbolic execution`。
- 已存在 manual proof 文件不被覆盖是正常行为。

### Step 5: 编译并修复 proof

进入 `SeparationLogic`：

```powershell
cd SeparationLogic
make <case_goal_check.vo>
```

若失败，按顺序处理：

1. 缺库或逻辑路径错误：检查 `_CoqProject`、`-slp`、目录重命名。
2. 语法错误：检查 `[| |]`、`“ ”`、字符串字面量、字段访问记号。
3. 空间谓词不匹配：检查 `data_at` 是否仍残留，或 strategy 是否未刷新。
4. hypothesis 编号错误：用当前 goal 中的 `PreH` / `H` 名称替换。
5. 未闭合目标：临时 `Show.` 查看剩余目标，修完后删除。

### Step 6: 字符串 case 的额外检查

字符串 case 更新后，推荐单独编译：

```powershell
make examples/LLM_bench/Engineering/string/strcat_goal_check.vo
make examples/LLM_bench/Engineering/string/strcmp_goal_check.vo
make examples/LLM_bench/Engineering/string/strcpy_goal_check.vo
make examples/LLM_bench/Engineering/string/strlen_goal_check.vo
```

如果失败：

- 先确认 `StringLib.v` 编译通过。
- 再看目标是否已经使用新版字符串表示。
- 最后修 manual proof，不要回退到旧数组展开式。

## 常见错误与处理

| 错误信息 / 现象 | 常见原因 | 处理方式 |
| --- | --- | --- |
| `Cannot find a physical path bound to logical path ...` | 目录重命名后 `_CoqProject` 或 `-slp` 未更新 | 同步物理路径和逻辑路径 |
| generated goal 大面积变化 | QCP / SAC 版本更新后的正常现象 | 重新运行 symexec 后修 manual proof |
| `Syntax error` 出现在纯断言附近 | 仍使用旧 `[| |]` 或引号编码错误 | 改为 `“ ... ”` 并保存为 UTF-8 |
| 空间谓词匹配失败 | `data_at` 与 `store` 新旧风格混用 | 迁移 annotation / strategy 到 `store` |
| 字符串 proof 失效 | 新 StringLib 改变了字符串字面量表示 | 使用 `StringLib` 中的规则重新证明 |
| `The reference H... was not found` | 新生成 goal 改变 hypothesis 编号 | 查看当前 context，改为当前 `H` / `PreH` |
| `Attempt to save an incomplete proof` | 自动化后仍有纯目标或空间目标 | 临时 `Show.`，补 rewrite / sep_apply / lia |

- [ ] 纯断言使用 `“ ... ”`，没有混用旧 `[| |]`。

- [ ] 相关 `*_goal_check.vo` 编译通过。
